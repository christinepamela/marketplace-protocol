## Part 1: Protocol Overview

### 1. What is Rangkai Protocol?

**Rangkai Protocol** is a decentralized marketplace protocol designed to globalize small businesses by connecting manufacturers and artisans with buyers through a **non-custodial, Bitcoin-native infrastructure**.

**Core Principles:**
- **Non-Custodial**: No platform holds funds; 7-day escrow with auto-release
- **Bitcoin-Native**: Lightning Network & on-chain settlement
- **Decentralized**: Federated search across multiple marketplaces
- **Privacy-First**: DID-based identity, optional KYC
- **Fair Fees**: 3% protocol fee, 0–5% client marketplace fee

### 8. Debugging Journey & Solutions

#### Phase 7 — Zero Issues 🎉

By following Phase 6 patterns exactly, Phase 7 completed in ~1 hour with only one minor issue:

**Quorum calculation:** Expected 2/3 but system uses 3/3.  
`Math.ceil(3/2) = 2` but the system enforces 3-of-3.  
**Fix**: Added a test for the third approval. Resolved in 5 minutes.

---

#### Phase 8 — Two Issues

**Issue 1: Field Name Mismatch**

**Symptom**: Events emitted but not received; logs showed `undefined` for `buyerDid`/`vendorDid`.  
**Root Cause**: `CreateOrderResponse` uses different field names than the `Order` type:
- `response.orderId` (not `response.id`)
- `response.buyerDid` doesn't exist in the response object

**Fix**: Map fields explicitly using request data:

```typescript
emitOrderEvent('order.created', {
  id: response.orderId,           // Map orderId → id
  orderNumber: response.orderNumber,
  buyerDid: buyerDid,             // From getUserDid(req)
  vendorDid: req.body.vendorDid,  // From request body
  status: response.status,
  total: response.total,
} as any);
```

**Time**: 30 minutes of debugging with `console.log`.

---

**Issue 2: TypeScript Null Checks**

**Symptom**: Compilation error — `'updatedOrder' is possibly 'null'`.  
**Root Cause**: TypeScript strict null checking on `getOrderById()` return type.  
**Fix**: Non-null assertion operator on event emissions:

```typescript
id: updatedOrder!.id,  // Safe because null was already checked earlier
```

Applied across 5 event emissions × 6 fields = 30 assertions.  
**Time**: 10 minutes.

### 13. Best Practices & Patterns

#### Critical Code Patterns

**Import vs Call:**
```typescript
// ✅ CORRECT
import { authenticate } from '../middleware/auth.middleware';
router.post('/', authenticate(), ...)

// ❌ WRONG
router.post('/', authenticate, ...)  // Missing ()
```

**Route order:**
```typescript
// ✅ CORRECT — specific routes first
router.get('/stats', ...)
router.get('/:id', ...)

// ❌ WRONG — parameterized routes first
router.get('/:id', ...)   // Would match '/stats'
router.get('/stats', ...)
```

**Event emission pattern:**
```typescript
// ✅ CORRECT — fetch fresh data before emitting
await service.operation();
const updated = await service.getById(id);
emitEvent('event.name', updated);

// ❌ WRONG — stale data
await service.operation();
emitEvent('event.name', {...});
```

**Error handling:**
```typescript
// ✅ CORRECT
try {
  // ...
} catch (error) {
  next(error);  // Pass to error middleware
}

// ❌ WRONG
} catch (error) {
  res.status(500).json({error});  // Handle inline
}
```

**Type safety:**
```typescript
// ✅ CORRECT
const userDid = getUserDid(req);            // Helper function
const order: Order = await service.getOrder(id);

// ❌ WRONG
const userDid = req.user.did;              // Might be undefined
```

#### Testing Patterns

- **Setup**: create test data first, generate auth tokens, store IDs for cleanup
- **Tests**: happy path → forbidden access → validation errors → not found errors
- **Cleanup**: delete in reverse dependency order, always run even if tests fail

#### Git Practices

```bash
git commit -m "feat(phase7): Add governance routes with 3-of-3 multisig

- 5 endpoints for proposal management
- 9 integration tests
- Public transparency features

Tests: 9/9 passing"
```

Commit after each phase. Never commit broken code. Run all tests before committing.
