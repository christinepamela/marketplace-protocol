## 🐛 OUTSTANDING PROBLEMS FOR SESSION 25

### Problem 1: Auto-Confirming Payment (CRITICAL)
**Symptom:** Every new order immediately shows "Payment Confirmed! 30652 confirmations"
**Root Cause:** ALL addresses in `bitcoin_payment_addresses` table are `confirmed = TRUE` because they have real testnet transactions on the blockchain from previous test sessions. The HD wallet derives addresses sequentially (index 0, 1, 2, 3...) and ALL existing indexes are burned.
**What was tried:**
- Deleted specific order records from `bitcoin_payment_addresses` — didn't help because backend reuses confirmed addresses
- Reset order status in `orders` table — didn't help
- Created new orders — still auto-confirms because new orders get assigned already-confirmed address indexes

**What needs to happen:**
The HD wallet needs to start from a fresh index that has NO blockchain history. Current max index is 3. Index 4 should be clean. The problem is the `generate-address` route reuses existing addresses for the same order. Since the DB record was deleted, it generates a NEW address at the next index — but the double-call bug (see Problem 2) is creating conflicting records.

**Verification steps:**
1. Go to Supabase → SQL Editor
2. Run: `SELECT * FROM bitcoin_payment_addresses ORDER BY derivation_index DESC;`
3. Check what the highest `derivation_index` is
4. The next new address (index N+1) should have no blockchain history
5. Verify on: https://blockstream.info/testnet/address/[NEW_ADDRESS]

### Problem 2: Double API Call (CRITICAL)
**Symptom:** Server logs show `POST /generate-address` being called TWICE every time
```
POST /generate-address 200 607ms
POST /generate-address 200 624ms  ← duplicate
```
**Root Cause:** `BitcoinPaymentFlow.tsx` calls `generatePaymentAddress()` inside a `useEffect` that fires twice in React StrictMode (development mode double-invocation). This creates a race condition.
**Fix needed in `BitcoinPaymentFlow.tsx`:**
Add a ref guard to prevent double execution:
```typescript
const hasGeneratedRef = useRef(false)

useEffect(() => {
  if (hasGeneratedRef.current) return
  hasGeneratedRef.current = true
  generatePaymentAddress()
  
  return () => {
    if (pollIntervalRef.current) {
      clearInterval(pollIntervalRef.current)
    }
  }
}, [])
```

### Problem 3: Vendor Not Updating After Payment (CRITICAL)
**Symptom:** Order shows "Payment Confirmed" on buyer side but vendor still sees "Payment Pending"
**Root Cause:** The order status update in `bitcoin.routes.ts` is failing silently. The `orders` table has `payment_status` with allowed values: `pending, processing, completed, failed, refunded, partially_refunded`. The code was setting it to `'confirmed'` which violates the constraint — the update fails silently.
**Fix needed in `bitcoin.routes.ts`** (the status update block):
```typescript
// WRONG:
payment_status: 'confirmed',

// CORRECT:
payment_status: 'completed',
```

### Problem 4: CoinGecko Rate Limiting (429 errors)
**Symptom:** `Error fetching Bitcoin price: AxiosError: Request failed with status code 429`
**Root Cause:** CoinGecko free API has rate limits. The double-call bug (Problem 2) is hitting the API twice per "Pay Now" click, and the 30-second polling also calls it. We're exceeding the free tier limit.
**Fix needed in `bitcoin.service.ts`:** Cache the BTC price for 60 seconds:
```typescript
private cachedBtcPrice: number | null = null
private cachedBtcPriceAt: number = 0
private readonly PRICE_CACHE_TTL = 60000 // 60 seconds

private async getBitcoinPrice(): Promise<number> {
  const now = Date.now()
  if (this.cachedBtcPrice && (now - this.cachedBtcPriceAt) < this.PRICE_CACHE_TTL) {
    return this.cachedBtcPrice
  }
  try {
    const response = await axios.get('https://api.coingecko.com/api/v3/simple/price', {
      params: { ids: 'bitcoin', vs_currencies: 'usd' }
    })
    this.cachedBtcPrice = response.data.bitcoin.usd
    this.cachedBtcPriceAt = now
    return this.cachedBtcPrice
  } catch (error) {
    console.error('Error fetching Bitcoin price:', error)
    return this.cachedBtcPrice || 50000 // Use cached or fallback
  }
}
```

### Problem 5: Page Auto-Refreshing Aggressively
**Symptom:** Order detail page polls every ~3 seconds making it hard to interact
**Root Cause:** Order detail page has a polling interval. It's normal but the interval may be too short.
**Not critical** — lower priority than Problems 1-4.
