# Why Rangkai Exists

Pam wrote about this in her 2025 Medium piece **"When Global Trade Leaves Small Businesses Behind"** (https://medium.com/@christinepamela/when-global-trade-leaves-small-businesses-behind-d9e31b6c292a). It's the canonical statement of purpose. Read it before doing any architecture work.

The short version: global trade infrastructure was built for large corporations. A Penang batik weaver, a Sabah honey producer, a Klang furniture maker — they exist, they make excellent products, they have buyers waiting in Berlin and Toronto, but they're functionally invisible because:

- They don't know how to navigate Incoterms, HS codes, customs forms
- They can't afford freight forwarders
- Their countries' banking infrastructure makes cross-border payments slow and expensive
- They have no platform built for *their* scale that connects to *international* buyers
- The platforms that exist (Alibaba, Amazon) extract too much and surface them poorly

**Rangkai is infrastructure for them.** Not a marketplace — a protocol that enables marketplaces. Anyone with vision and a small budget can spin up a Rangkai-powered marketplace serving their country's small businesses. The federation is the point: a buyer on a Malaysian Rangkai marketplace finds an Indonesian seller through their local Indonesian Rangkai marketplace, with the protocol coordinating identity, payments, logistics, and trust across both.

This shapes every architecture decision:
- Why the protocol takes only 0.5% (small businesses can't afford more)
- Why we use Bitcoin (banking infrastructure exclusion is the problem)
- Why we leverage Nostr (sovereign identity without big-platform dependence)
- Why logistics is its own pool (small vendors can't negotiate freight individually)
- Why search must federate (small vendors aren't visible without cross-marketplace discovery)
- Why we don't store passwords or KYC documents at the protocol level (we're not the government's chopping board)

When in doubt, re-read the Medium piece. The technical decisions follow from the purpose.

# Architectural Principles (Across All Versions)

These hold regardless of v1/v2/v3 — they're the constants. New work must preserve all of them.

## 1. The protocol is passive infrastructure

The protocol provides rails. Marketplaces operate. The protocol never enforces policy (only the rails). This is why:
- 0.5% from seller + 0.5% from logistics, never higher
- No KYC at the protocol layer (marketplaces handle per jurisdiction)
- No content moderation at the protocol (marketplaces moderate)
- No transaction blocking (protocol records what happened; marketplaces decide what to allow)

## 2. Marketplace-marketplace federation is the goal, from day one

A Malaysian Rangkai marketplace must be able to discover and trade with an Indonesian Rangkai marketplace. This is not v2 — it's the *reason the protocol exists*. Every design must preserve federation:

- Identity layer must work across marketplaces (DID-based, not marketplace-specific accounts)
- Search must extend across marketplaces (not just within one)
- Trust signals must port (a vendor's reputation moves with them)
- Logistics is one pool shared across marketplaces
- Payment rails must work for any marketplace

**v1 is "one marketplace done well, designed to federate." Not "one marketplace, federation later."** The difference is everywhere in the schema and the API design.

## 3. Cryptographic identity (Pam owns nothing she doesn't need)

Identity is keypairs (secp256k1 — same curve as Bitcoin and Nostr). Marketplaces hold credentials per their needs. Protocol never holds passwords, private keys, or KYC documents. This trades a small amount of convenience for genuine sovereignty.

## 4. Leverage existing ecosystems without becoming dependent

Nostr for messaging transport. Bitcoin for permissionless payments. Standard cryptographic primitives. But Rangkai owns the search experience, the marketplace UX, the federation rules, the trust layer. We use Nostr as plumbing; we don't ask buyers to "use Nostr" — they use Rangkai which happens to use Nostr underneath.

## 5. Small business first

UX simplicity over technical purity. A Penang batik weaver should be able to list her products without knowing what an HS code is (we suggest one). A buyer should find vendors without knowing about Nostr relays. Hide complexity, deliver capability.

## Bug 4: Order showed "PAID" without any Bitcoin being sent

This is the rabbit hole that produced the rest of the session.

**Test:** New vendor (testBit) created new product ("bitcoin leather" $15). Buyer (Jane) added to cart, selected Bitcoin on-chain, placed order ORD-2026-153612-4VX. **QR code briefly appeared, then disappeared.** Vendor dashboard immediately showed the order as `paid`.

**Database state confirmed via SQL:**
- `orders` row: `status=paid`, `payment_status=completed`
- `bitcoin_payment_addresses` row: address `tb1qnpzzqjzet8gd5gl8l6gzhuc4s9xv0djt99y09w`, `confirmed=true`, `confirmations=159808`, `amount_received=0`
- `escrows` row: `status=held`, `held_at` set

**The cause:** The address assigned to the new order was already used in past testing. It has **159,808 confirmations on testnet from months ago**, with `amount_received=0` for THIS order. The code in `bitcoin.service.ts` saw `confirmations >= 3` and flipped the order to paid without checking that money was received for THIS payment.

This bug was documented in Session 26 handover's troubleshooting section:
> Bitcoin Payment Auto-Confirms Immediately
> Cause: The address assigned to the order already has blockchain history (burned index).

The bug was documented but never fixed. The `getNextDerivationIndex()` function in `bitcoin.service.ts` derives the next BIP84 address from the database's record of highest existing index. After Supabase data was wiped/reset at some point, the table was empty, so it returned index 0 — which is burned from prior testing.

## Bug 5 (the actual root cause): No login flow exists in the protocol

This is what the cascade led to. While diagnosing why testBit was burning new derivation indexes, Pam mentioned: *"every time I setup a new vendor and product and if I refresh, it's gone. and even if I log in back to that username, it's not there anymore."*

**Verified via SQL:**
- `identities` table has **33 distinct vendor DIDs** in the products table (one with 17 products, one with 2, one with 2, 30 others with 1 each)
- `testBit` only exists once today — confirmed Pam didn't re-register testBit specifically
- But across months, every "login" has been a re-register, creating new identity DIDs

**Read `identity.service.ts`** — it has `registerIdentity()`, `getIdentityByDID()`, `updateProfile()`, `verifyIdentity()`. It has **NO** `login()`, **NO** password verification, **NO** bcrypt or hash anywhere. `findstr /s /n "signUp\|signIn\|bcrypt\|hash" src\core\layer0-identity\*.ts` returned empty.

**The marketplace's register page already collects email + password fields** — but the protocol's register schema only accepts `displayName/country/businessType/avatarUrl/bio`. The marketplace was collecting passwords and **silently throwing them away** before sending to the protocol.

**This is B2 in tech debt.** It's not a small bug. It's the foundational gap that explains:
- Why orders show "paid" without payment (burned indexes from session-to-session)
- Why the database has 33 ghost vendor DIDs
- Why every Session 26 test user (`vendor.test@rangkai.local`, etc.) is gone
- Why every test session feels like starting from scratch

Without a real login, every test session creates new identities. Old identities still exist (data persists), but they're orphaned because the new session's DID doesn't own anything they created.

## What we decided about auth (the meat of the session)

After establishing the gap, we spent significant time designing the auth architecture properly rather than slapping in a quick fix. Pam's privacy stance shaped every decision.

### Architecture decisions

**Three identity types stay** (`kyc | nostr | anon`) but reframed: they describe TRUST POSTURE, not technology. All three use the same secp256k1 keys underneath (same curve as Bitcoin and Nostr).

- **KYC** — marketplace stores verified business info, plus password hash. Marketplace optionally participates in future Frostr 2-of-3 setup. KYC verification is the marketplace's job. Documents flow through marketplace, never through protocol.
- **Nostr** — user brings their existing nsec OR generates a fresh Rangkai key. User can choose to publish npub publicly. Same key tech, different social posture.
- **Anon** — fresh keys, no marketplace storage of personal info, BTC payments only, no recovery.

**The protocol's job is small:**
- Store DIDs and their `type` flag
- Verify signatures
- Issue session JWTs after successful signature challenge
- **Never** store passwords, private keys, or KYC documents

**The marketplace's job is everything else:**
- KYC verification (jurisdictional)
- Password UX (Pattern 1: locally encrypted key + password)
- Frostr ceremony (future)
- Dispute mediation

### Why same curve as Nostr/Bitcoin

secp256k1 is the curve Bitcoin uses, Nostr uses, Ethereum uses. By choosing it:
- Users can OPTIONALLY use one keypair for Nostr identity AND Rangkai identity (their choice, not forced)
- Existing Nostr clients can display Rangkai users' profiles (if they choose to publish)
- Future hardware wallet support becomes possible
- We're not inventing crypto — we're using primitives with 15+ years of peer review

### Default: fresh keys per user, NOT shared with Nostr

**Important nuance.** Pam was initially attracted to "same keypair for Nostr and Rangkai." We agreed the right default is **generate a fresh Rangkai key** at registration. Offer "import existing nsec" as an option for users who explicitly want unified identity.

Why: privacy correlation. If `npub_xyz` is a Rangkai vendor AND has a public Nostr profile with politically sensitive posts, those are now linked. A subpoena against Nostr activity finds Rangkai transactions. **One key = one observable identity across networks.** Some users want this. Many don't. Default is privacy-preserving (separate keys); unified is opt-in.

### Password recovery: Frostr is the path, but not in v1

Pam asked the right question: *"if they lose their password how do they get it back with pattern 1?"*

Honest answer: **for non-custodial key-based auth, there is no perfect recovery.** The platform doesn't have the key.

Options:
1. **BIP39 backup phrase** — show 12 words at registration. Users save them. Practical recovery rate is dismal (~20%) because users don't actually save them.
2. **Cloud backup with encryption** — encrypted-with-password key copy to iCloud/Drive. Doesn't help if password is forgotten.
3. **Social recovery** (Argent/Safe style) — designated guardians. Mature but UX-heavy.
4. **Custodial fallback** — trade sovereignty for safety. Some users want this.
5. **Frostr / FROST threshold signing** — split nsec into k-of-n shares. Lose one share, rotate it out. Pomade (FROST + email recovery) and Pomegranate (FROST + Google SSO) are alpha-released in 2025-2026.

Frostr is the architecturally right answer. It's also alpha. Don't depend on it for v1.

**v1 decision:** KYC users get marketplace-mediated password reset (standard email link). Anon/Nostr users get BIP39 backup phrase shown once at registration with strong warnings. **v2 roadmap: Frostr-style 2-of-3 with marketplace + recovery service participation.**

### Bitkey framing for explaining Frostr to non-technical users

Pam asked if we can simplify how Frostr is explained. Bitkey's current framing is excellent — steal it directly:

> Your account is protected by three keys. One lives in your browser, one with this marketplace, and one with a recovery service of your choice. Any two of them can sign you in. Forgot your password? The marketplace and your recovery service can issue you a new browser key. Marketplace shuts down? Your browser and recovery service still let you log in elsewhere. The marketplace alone can never sign in as you.

Visual: three key icons. Two need to be active to sign in. Show the marketplace key struck out and the other two cooperating — that's the "marketplace can't impersonate you" message.

**The data model in v1 anticipates this even though v1 doesn't implement it.** Adding a `signing_strategy` column with default `single_key` and future values `frostr_2of3` etc. means the v2 migration is additive, not a rewrite.

### Messaging architecture

**v1:** NIP-44 encrypted direct messages between order participants only. Scoped to orders (every conversation has an `order_id`). Marketplace stores message references and encrypted blobs.

**Disputes (Pattern 2 — UPDATED to unilateral disclosure with notification):**

The original "both consent or neither wins" pattern fails when a bad actor refuses to consent. Pam caught this: *"in the event one is a con artist who knows and does not reveal the details."*

**Refined design:**
- Either party can submit the decrypted thread to the dispute resolver. Both parties have the technical ability to decrypt (NIP-44 uses ECDH shared secrets — both parties always could decrypt all messages in their thread; "consent" is policy, not crypto)
- The submitting party stakes their own privacy too — their messages are exposed alongside the other party's
- The other party is notified that disclosure has happened
- The other party can submit additional context or counter-evidence (their own decryptions of the same thread, additional messages, supporting documents)
- Refusing to engage with the dispute weighs against them in the dispute logic
- Marketplace **never** has unilateral surveillance access

This matches how real-world commercial dispute resolution works: a text someone sent you is admissible without their consent. They can object to context, not to the existence of the message. Privacy expectation is "this won't be public," not "you can never use this against me in a dispute."

Solves the con artist problem: honest party reveals their copy, bad actor's refusal to engage IS evidence.

### Messaging v2 (before soft launch — Pam pushed back, accepted)

**Public Rangkai channels.** Not generic Nostr soup (Pam: *"if users have to just general nostr than I think it's a downfall. it's too centralised. I have been a nostr user for 3 years and even I get annoyed."*).

Rangkai-curated channels for:
- Seller-to-seller community (advice, sourcing tips, customs experience)
- Buyer reviews of vendors
- Marketplace operator announcements
- Regional/category-specific groups (e.g., "Malaysian craft exporters")

Discovery and moderation are Rangkai-specific. Underlying transport may be Nostr relays but the UI and indexing are Rangkai's.

**Pam will have deeper conversations about this in Session 29+** before any v2 messaging work begins. Architecture not finalized.

### Search architecture — THIS IS THE PRODUCT, not auth

Critical reframing from Session 28 end.

Pam pushed back: *"because these are small business vendors who are generally not publicly known in the world, it's incredibly important that it's easy to locate them on the rangkai protocol... if I were to buy a product from my country's sellers but not available I can search and find from another marketplace in another country. that's the goal."*

**Search is THE feature.** Not auth, not Bitcoin payments. Search is what makes Rangkai useful. Without it, a Penang batik weaver may as well not exist to a Berlin furniture buyer.

Pam's observation: Nostr search is terrible. She, a 3-year Nostr user, can't find her own past posts. We cannot make buyers depend on Nostr search to find vendors.

**v1 search:** Centralised Postgres search inside the protocol. Works for a single Rangkai marketplace today. This is what currently exists.

**v2 search (before soft launch):** Real federated search across multiple Rangkai marketplaces. Same product visible whether the buyer is on marketplace-Malaysia or marketplace-Germany. Nostr-event-based product publication is one architectural option; a Rangkai-specific federation protocol is another. **Pam will have deeper conversation in Session 29+** about the right approach.

**Whatever the federation model: Rangkai must own the search index quality.** Use Nostr as publication/transport if it makes sense, but the search experience must be Rangkai-defined. Buyers should be able to find Malaysian batik weavers from Berlin without knowing Nostr exists.

**v3:** Search filters by Incoterm, HS code category, certifications, ESG, lead time, MOQ. Visual search by image. Translation (vendor lists in Malay, buyer searches in German).

### Marketplaces as serverless entrepreneurs

Pam: *"Also because it's leveraging Nostr, entrepreneurs who setup marketplaces can now enjoy a more serverless benefit / option."*

The vision: a marketplace entrepreneur in Indonesia can spin up a Rangkai-powered storefront with minimal infrastructure. Identity layer is cryptographic (no auth server). Messaging is over public Nostr relays (no chat server). Search federates across other Rangkai marketplaces. Payment rails are Stripe (when KYC) and Bitcoin on-chain (when not). The only thing the marketplace truly needs to operate is a small database for KYC docs (per-jurisdiction requirement) and a static frontend.

### Nostr data load — design concern raised S28

Pam noted: *"nostr has large data all the time. even when i run primal my laptop is noisy."*

This is real. Nostr clients sync large amounts of historical data. For Rangkai users, we must:
- **Subscribe scopes:** Rangkai clients only subscribe to relays for specific kinds (order chats, future product publication events), never to the general Nostr firehose
- **Filtered relays:** consider running Rangkai-specific relays that only carry Rangkai-relevant kinds (v2/v3)
- **Server-side aggregation:** marketplace caches Nostr data and serves to client. Client doesn't directly hit relays.
- **Lightweight protocol:** if we use Nostr for product publication, define kinds tightly so clients fetch only what they need

Added to tech debt as D19. Cross-cutting design concern that affects messaging (v1) and search federation (v2).

# Troubleshooting Reference

## API server won't start

**Symptom:** `Router.use() requires a middleware function but got a Object` or similar.

**Cause:** A route file imported in `src/api/routes/index.ts` is empty or has wrong export.

**Fix:**
- Check the line number in the error — counts from line 1 of `routes/index.ts`
- The file that line references is the problem
- For empty files: remove the import + mount from `routes/index.ts`
- For files with wrong export: ensure `export default router;` at the bottom

## "TypeError: fetch failed" from scheduler on startup

**Symptom:** API boots but immediately logs scheduler errors:
```
[Scheduler] Escrow auto-release job failed: TypeError: fetch failed
[Scheduler] Proposal expiry job failed: TypeError: fetch failed
[Scheduler] Bitcoin monitor query failed: TypeError: fetch failed
```

**Cause:** Supabase project is paused.

**Fix:** Go to https://supabase.com/dashboard. Find Rangkai project. Click Restore. Wait ~30 seconds. Scheduler errors should stop on next interval (60s for Bitcoin monitor, 5 min for others).

## Bitcoin payment auto-confirms immediately

**Cause:** Burned derivation index. The address has prior testnet history.

**Diagnostic:**
```sql
SELECT derivation_index, address, confirmed, confirmations, amount_received 
FROM bitcoin_payment_addresses 
ORDER BY derivation_index ASC;
```

**Fix (manual, temporary):** Insert placeholder rows for indexes 0-N where N is the last known burned index. Per Session 26, indexes 0-4 were burned. Per Session 28, the table was emptied so index 0 was reused.

**Real fix:** Implement burned-index detection (B10 in tech debt) — `getNextDerivationIndex` should check Blockstream before assigning.

## Vendor logs in, can't see their products

**Cause:** B2 (no login flow). What looks like "logging in" is actually re-registering, creating a new DID. The products are still in the database, owned by the original DID, but the new DID has nothing.

**Diagnostic:**
```sql
-- Find all identities matching a name
SELECT did, public_profile->>'displayName' as name, created_at
FROM identities
WHERE public_profile->>'displayName' = 'testBit'
ORDER BY created_at DESC;
```

If multiple rows: ghost vendor pattern confirmed.

**Fix:** Build real login (Session 29 plan above).

## Port conflict on 3000

**Cause:** Next.js bound to 3000 before API server did.

**Fix:** `rangkai-marketplace/package.json` "dev" script must include `-p 3001`. If it's missing, add it. This was fixed in S28.

## Stripe webhook signature failures

**Cause:** Stripe CLI is forwarding to wrong path, OR webhook secret in `.env` doesn't match Stripe CLI's signing secret.

**Fix:** Restart Stripe CLI. Copy the new `whsec_...` signing secret from CLI output to `.env`. Restart API server.
