# Why Rangkai Exists

Pam wrote about this in her 2025 Medium piece **"When Global Trade Leaves Small Businesses Behind"** (https://medium.com/@christinepamela/when-global-trade-leaves-small-businesses-behind-d9e31b6c292a). Read it before doing any architecture work. The technical decisions follow from the purpose.

Short version: global trade infrastructure was built for large corporations. A Penang batik weaver, a Sabah honey producer, a Klang furniture maker — they exist, they make excellent products, they have buyers waiting in Berlin and Toronto, but they're functionally invisible. Rangkai is infrastructure for them. Not a marketplace — a protocol that enables marketplaces. Federation is the point.

# Architectural Principles (Always-On)

1. **The protocol is passive infrastructure** — provides rails, never enforces policy. 0.5% fee from seller + 0.5% from logistics, never higher. No KYC at protocol layer.
2. **Marketplace-marketplace federation is the goal, from day one** — not v2, the reason the protocol exists. Every design must preserve the federation path. v1 is "one marketplace done well, designed to federate."
3. **Cryptographic identity** — secp256k1 keypairs (same curve as Bitcoin and Nostr). Protocol never holds passwords, private keys, or KYC documents.
4. **Leverage existing ecosystems without becoming dependent** — Nostr is plumbing, Rangkai owns UX and search.
5. **Small business first** — hide complexity, deliver capability.

# Troubleshooting Reference

## API server won't start
**Symptom:** `Router.use() requires a middleware function but got a Object`
**Cause:** A route file in `routes/index.ts` is empty or has wrong export.
**Fix:** Check the line number, find the offending file, remove or fix the mount.

## "TypeError: fetch failed" on startup
**Cause:** Supabase is paused.
**Fix:** Go to supabase.com/dashboard, restore project, wait 30s.

## Bitcoin 429 errors from Blockstream
**Cause:** Too many requests — 700 req/hour free tier per IP.
**Fix (immediate):** Wait 1 hour. Scheduler retries automatically.
**Fix (permanent):** B12 makes 429 graceful. R10 (BTCPay) eliminates polling entirely.
**Important:** Do NOT use the `abandon abandon...` mnemonic. Every address derived from it is burned. Always generate fresh mnemonics.

## Mnemonic rotation (if needed again)
1. Generate fresh: `node -e "const bip39 = require('bip39'); console.log(bip39.generateMnemonic(256));"`
2. Replace `BITCOIN_MNEMONIC` in `.env` with new 24 words
3. Set `BITCOIN_START_INDEX=0`
4. Restart API server — burn scan runs automatically
5. Clean up placeholder rows after scan completes:
   ```sql
   DELETE FROM bitcoin_payment_addresses WHERE order_id LIKE 'burned-placeholder-%';
   ```
6. Store mnemonic securely offline — for production/mainnet this means paper backup or hardware wallet seed storage, never plain text in version control or cloud notes.

## Burned placeholder rows in bitcoin_payment_addresses
When the burn scan runs, it writes rows with `order_id LIKE 'burned-placeholder-%'`. These are harmless but pollute queries. Safe to delete at any time:
```sql
DELETE FROM bitcoin_payment_addresses WHERE order_id LIKE 'burned-placeholder-%';
```
Real order rows have UUID `order_id` values and are never affected by this query.

## Bitcoin payment auto-confirms immediately
**Was:** Burned derivation index (fixed S29 — B10).
**If it happens again:** Check `.env` has the new mnemonic (not `abandon...`) and `BITCOIN_START_INDEX=0`. Check `bitcoin_payment_addresses` — if `confirmed=true` and `amount_received=0`, a burned index was assigned.

## Vendor logs in, can't see their products
**Was:** B2 — login was a re-register, creating a new DID (fixed S29).
**If it happens again:** Run `SELECT did, email FROM identities WHERE email = 'their@email.com'` — should be one row.

## Port conflict on 3000
**Cause:** Next.js bound to 3000 before API server.
**Fix:** `rangkai-marketplace/package.json` "dev" must be `next dev -p 3001`. Fixed S28.
