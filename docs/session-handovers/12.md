## 🏛️ PROTOCOL ARCHITECTURE

### Layer Structure (7 Layers)

```
Layer 0: Identity & Reputation (DID, KYC, Nostr, Anonymous)
Layer 1: Discovery & Catalog (Products, Search, Categories)
Layer 2: Order Management (Orders, Payments, Escrow)
Layer 3: Logistics & Fulfillment (Shipping, Tracking, Providers)
Layer 4: Dispute Resolution (Claims, Evidence, Mediation)
Layer 5: Analytics & Insights (Stats, Reports, Recommendations)
Layer 6: Governance (Voting, Proposals, Protocol Upgrades)
```

### Core Components

**Identity System (Layer 0)**
- DID-based authentication
- Multiple identity types: KYC, Nostr, Anonymous
- JWT tokens with 30-day refresh tokens
- Permission-based access control

**Catalog System (Layer 1)**
- Product CRUD operations
- Federated search across marketplaces
- Category/subcategory taxonomy
- Product stats tracking (views, inquiries, orders)

**Order System (Layer 2)**
- Order lifecycle management
- Escrow payment system
- Order status tracking
- Buyer/vendor order views

**Logistics System (Layer 3)**
- Multiple logistics provider support
- Real-time tracking updates
- Shipping quotes
- Delivery confirmation

**Dispute System (Layer 4)**
- Claim filing and management
- Evidence upload
- Mediation workflow
- Arbitration system

### SDK Structure

```
marketplace-protocol/packages/sdk/
├── src/
│   ├── client.ts          # HTTP client with auth
│   ├── types.ts           # All TypeScript types
│   ├── errors.ts          # Error classes
│   └── modules/
│       ├── identity.ts    # Layer 0: Auth, registration
│       ├── catalog.ts     # Layer 1: Products, search
│       ├── orders.ts      # Layer 2: Order management
│       ├── logistics.ts   # Layer 3: Shipping, tracking
│       ├── disputes.ts    # Layer 4: Dispute resolution
│       ├── analytics.ts   # Layer 5: Stats, insights
│       └── governance.ts  # Layer 6: Voting, proposals
```

### API Route Format

**Pattern:** `/api/v1/{layer}/{resource}/{id?}/{action?}`

**Examples:**
- `POST /api/v1/identity/register` - Register new user
- `GET /api/v1/catalog/products?vendorDid=xxx` - Get vendor products
- `POST /api/v1/orders/{id}/ship` - Mark order as shipped
- `GET /api/v1/logistics/providers` - Get logistics providers
- `POST /api/v1/disputes/{id}/submit-evidence` - Submit dispute evidence

### Database Schema

**Supabase Tables:**
- `identities` - User profiles and DIDs
- `products` - Product catalog
- `orders` - Order records
- `order_items` - Order line items
- `shipments` - Shipping information
- `tracking_events` - Shipment tracking
- `disputes` - Dispute cases
- `dispute_evidence` - Evidence files
- `analytics_events` - User activity tracking

### Component Patterns

**Page Components** (`app/*/page.tsx`):
- `'use client'` directive (interactive pages)
- Auth checks with `useAuth()` hook
- Data loading in `useEffect`
- Error handling with try/catch
- Loading states with spinners

**Reusable Components** (`components/*`):
- TypeScript interfaces for props
- Prop validation
- Consistent styling (Studio McGee theme)
- Accessibility considerations

**Forms:**
- Controlled inputs with React state
- Validation before submission
- Inline error messages
- Loading states on submit buttons
- Success/error alerts

### Authentication Flow

1. User visits `/auth/register`
2. Selects identity type (KYC/Nostr/Anonymous)
3. SDK creates identity + JWT token
4. Token stored in localStorage: `rangkai_token`
5. Token auto-refreshed every 30 days
6. AuthContext provides user state globally

### Data Flow

```
User Action
    ↓
Component Event Handler
    ↓
SDK Method Call
    ↓
HTTP Request to Protocol
    ↓
Protocol Processes Request
    ↓
Database Operation
    ↓
Response to SDK
    ↓
Component Updates State
    ↓
UI Re-renders
```

## ✅ BEST PRACTICES & GUIDELINES

### Development Workflow

**1. Zero Assumptions Rule**
- **ALWAYS** request files before making changes
- Never assume file structure or content
- Ask for confirmation when unsure

**2. File Request Format**
```
Before building/modifying, request:
1. Current file (if modifying)
2. Related files (imports, dependencies)
3. Type definitions (if using SDK types)
```

**3. Testing Before Committing**
- Test all CRUD operations manually
- Run relevant protocol layer tests
- Check browser console for errors
- Verify no broken features

**4. Token Management**
- Check usage every major section
- Write handover at 60% tokens
- Hard stop at 80% tokens
- Update tracking in handover doc

**5. Git Commits**
- Commit after each complete phase
- Write detailed commit messages
- Include: features, fixes, testing, next steps
- Never commit broken code

### Code Standards

**TypeScript:**
```typescript
// ✅ DO: Explicit types
interface Product {
  id: string;
  name: string;
  price: Price;
}

// ❌ DON'T: Implicit any
function processProduct(product) {
  // TypeScript error
}

// ✅ DO: Type SDK responses
const products: Product[] = await sdk.catalog.getByVendor(did)

// ❌ DON'T: Untyped SDK calls
const products = await sdk.catalog.getByVendor(did)
```

**React Components:**
```typescript
// ✅ DO: Props interface
interface ButtonProps {
  onClick: () => void;
  children: React.ReactNode;
  loading?: boolean;
}

export default function Button({ onClick, children, loading }: ButtonProps) {
  return (
    <button onClick={onClick} disabled={loading}>
      {loading ? 'Loading...' : children}
    </button>
  );
}

// ❌ DON'T: No types
export default function Button(props) {
  return <button {...props} />
}
```

**Error Handling:**
```typescript
// ✅ DO: Comprehensive error handling
try {
  const result = await sdk.orders.create(orderData);
  alert('Order created successfully!');
  router.push(`/orders/${result.orderId}`);
} catch (error: any) {
  console.error('Failed to create order:', error);
  alert(error.message || 'Failed to create order');

// ❌ DON'T: Silent failures
const result = await sdk.orders.create(orderData);
router.push(`/orders/${result.orderId}`);
```

**SDK Usage:**
```typescript
// ✅ DO: Use SDK methods
const products = await sdk.catalog.getByVendor(vendorDid);

// ❌ DON'T: Direct fetch() calls
const response = await fetch('http://localhost:3000/api/v1/catalog/products?vendorDid=xxx');
const products = await response.json();

// Why? SDK handles:
// - Authentication headers
// - Error mapping
// - Type safety
// - Response formatting
```

**State Management:**
```typescript
// ✅ DO: Zustand for global state (cart, user prefs)

const { items, addItem } = useCartStore();

```

### Do's and Don'ts

**✅ DO:**
- Use TypeScript strict mode
- Follow Studio McGee color palette
- Use SDK for all API calls
- Handle loading states
- Show error messages to users
- Validate form inputs
- Use semantic HTML
- Add accessibility attributes
- Test on mobile viewport
- Update handover document continuously

**❌ DON'T:**
- Modify protocol without testing
- Skip error handling
- Use magic numbers (define constants)
- Hardcode API URLs
- Store sensitive data in localStorage (except JWT)
- Use inline styles (use Tailwind)
- Commit commented-out code
- Leave console.logs in production
- Break backward compatibility
- Assume file locations
